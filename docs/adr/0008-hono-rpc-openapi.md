# Hono RPC + hono-openapi as the single source of truth for the API contract

Route definitions in `apps/api/src/routes/` are the single source of truth: `hono-openapi` derives the OpenAPI spec at runtime from the Hono route and Zod validator annotations, and `@hono/client` exposes a fully-typed RPC client consumed by the web app. There is no hand-authored OpenAPI spec file — `apps/api/openapi.json` is auto-generated by the `pnpm openapi:generate` script and committed. A CI check (`check-openapi.yaml`) fails the PR if the committed spec is out of sync with the code.

This is the opposite of a spec-first approach: the TypeScript types are authoritative and the spec follows. The benefit is that type drift between the API and its callers is impossible — changing a route's Zod schema immediately breaks the web app at compile time if callers aren't updated.

## Considered options

- **Spec-first (write OpenAPI, generate types)** — rejected because hand-maintaining a 3000-line JSON spec is error-prone and adds a step to every route change.
