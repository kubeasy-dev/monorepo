## Implementation Decisions

### Submit Route Refactoring
- Remove inline XP calculation, streak calculation, and all DB inserts for userXp/userXpTransaction from submit.ts (steps 11-16)
- Keep: objective validation, enrichment, submission DB insert, progress status update, SSE publish
- Dispatch a single `CHALLENGE_SUBMISSION` BullMQ job (single entry point for async pipeline)
- HTTP response changes to `{ success, objectives }` ‚Äî **no XP fields** (xpAwarded, totalXp, rank, rankUp removed)
- `better-all` replaces `Promise.all`/`Promise.allSettled` in submit.ts and workers

### BullMQ Worker Architecture
Each queue = one atomic operation:

**challenge-submission worker** (orchestrator role):
- Calculates `isFirstChallenge` from `userXpTransaction` count
- Calculates streak via `calculateStreak(userId)`
- Calculates XP amounts via `calculateXPGain({ difficulty, isFirstChallenge, currentStreak })`
- Fires PostHog `challenge_completed` analytics (trackChallengeCompletedServer)
- Dispatches `XP_AWARD` job for base XP (`action: "challenge_completed"`)
- If `isFirstChallenge`: dispatches a second `XP_AWARD` job (`action: "first_challenge"`)
- If `xpGain.streakBonus > 0`: dispatches a third `XP_AWARD` job (`action: "daily_streak"`)

**xp-award worker** (single atomic operation per dispatch):
- DB insert: `userXp` UPSERT (add xpAmount to totalXp) + `userXpTransaction` insert
- Publish generic cache-invalidation SSE event to Redis channel `events:{userId}` with `{ queryKey: ["user", "xp"] }` so browser invalidates its XP/rank query

**user-signin worker** (renamed from user-lifecycle, BullMQ Flow):
- Queue name in `@kubeasy/jobs` renamed `USER_SIGNIN` (was `USER_LIFECYCLE`)
- Implemented as a BullMQ Flow with parent + 3 child jobs:
  - Child 1: `identifyUserServer(userId, { email })` (PostHog identify)
  - Child 2: `createResendContact({ email, userId })` ‚Üí returns `contactId`
  - Child 3: `trackUserSignupServer(userId, provider, email)` (PostHog event)
- Parent job captures `contactId` from child 2 result and runs `db.update(user).set({ resendContactId: contactId })`
- `better-all` used where child jobs can run in parallel

### Generic Cache-Invalidation SSE (replaces ALL validation-specific SSE from Phase 5)
The Phase 5 `validation:{userId}:{challengeSlug}` channel and `/sse/validation/:slug` endpoint are superseded ‚Äî they did the same thing as the generic approach but limited to one event type.

**What Phase 5.1 removes:**
- `GET /api/sse/validation/:challengeSlug` endpoint (deleted from sse.ts)
- `redis.publish("validation:...")` call in submit.ts (replaced)
- `useValidationSSE` hook in apps/web (replaced by `useEventSSE`)

**What Phase 5.1 adds:**
- **Single endpoint**: `GET /api/sse/events` ‚Äî auth required, dedicated ioredis subscriber on `events:{userId}` per connection, same cleanup pattern (onAbort + 30s heartbeat)
- **Redis channel**: `events:{userId}` ‚Äî one channel per user, all event types multiplexed
- **Event payload shape**: `{ queryKey: string[] }` ‚Äî browser calls `queryClient.invalidateQueries({ queryKey })` on receipt, nothing else
- **Publish sites**:
  - `submit.ts` ‚Üí `{ queryKey: ["submissions", "latest", challengeSlug] }`
  - `xp-award worker` ‚Üí `{ queryKey: ["user", "xp"] }`
- **apps/web hook**: `useEventSSE(enabled)` ‚Äî subscribes to `GET /sse/events`, dispatches `invalidateQueries` for every received event. Replaces `useValidationSSE`.
- **ChallengeMission** updated to use `useEventSSE(status === "in_progress")` instead of `useValidationSSE`

### Onboarding API (feature parity with original Next.js)
**Schema changes** (drop Vercel-specific columns):
- Remove from `userOnboarding`: `workflowRunId`, `tokenWebhookUrl`, `loginWebhookUrl`, `setupWebhookUrl`, `startWebhookUrl`, `completeWebhookUrl`
- Keep: `completedAt`, `skippedAt`, `cliAuthenticated`, `clusterInitialized`

**New Hono routes** (mounted at `/api/onboarding`):
- `GET /api/onboarding` ‚Äî reads status, derives `currentStep` (1-7) from `userOnboarding` + `apikey` + `userProgress` tables (same logic as original `getStatus`). Uses `better-all` for parallel DB queries.
- `POST /api/onboarding/start` ‚Äî creates `userOnboarding` row (upsert), marks wizard started
- `POST /api/onboarding/complete` ‚Äî sets `completedAt`, fires `trackOnboardingCompletedServer`
- `POST /api/onboarding/skip` ‚Äî sets `skippedAt`, fires `trackOnboardingSkippedServer`

**CLI routes updated** (parit√© with original):
- `POST /api/cli/user` ‚Äî add: upsert `userOnboarding.cliAuthenticated = true` + call `trackCliLoginServer(userId, cliMetadata)`. Body: `{ cliVersion, os, arch }`.
- `POST /api/cli/track/setup` ‚Äî **new route** in Hono: upsert `userOnboarding.clusterInitialized = true` + call `trackCliSetupServer(userId, cliMetadata)`. Same contract as original.

**Challenge start** analytics:

## Specific Ideas

- "une queue -> une operation plus ou moins atomique mais c'est le seul point d'entr√©e" ‚Äî each BullMQ queue job = one atomic operation; challenge-submission is the single entry point after submit
- "les queues peuvent m√™me s'appeler entre elles" ‚Äî workers dispatch jobs to other queues (challenge-submission ‚Üí xp-award √ó N)
- Use `better-all` (https://github.com/shuding/better-all) everywhere instead of Promise.all/Promise.allSettled ‚Äî the onboarding GET handler already used it in the original app
- Submit HTTP response drops XP fields ‚Äî browser gets XP updates via the generic SSE cache-invalidation channel instead
- Generic SSE: one endpoint `GET /api/sse/events`, one Redis channel `events:{userId}`, payload is `{ queryKey }` only ‚Äî browser just calls `invalidateQueries(queryKey)`, no type discrimination needed
